## Users and Groups: Azure Identity and Access Management (IAM)

In Azure, identities are managed by **Microsoft Entra ID** (formerly Azure Active Directory) — a cloud-based identity provider that is separate from Azure resource management. RBAC assignments in Azure always reference an Entra ID principal.

### Principal Types

| Principal type | Identifier | Typical use |
|---|---|---|
| **User** | `alice@example.com` | Human operator — lives in Entra ID directory |
| **Group** | `sg-prod-admins` | Collection of users and/or service principals — assign RBAC to the group |
| **Service Principal** | App registration object ID | Application identity — programmatic access from external apps or pipelines |
| **Managed Identity** | System-assigned or user-assigned | Workload identity for Azure resources — no credentials to manage (preferred) |
| **Guest user** | `alice_external.com#EXT#@tenant.onmicrosoft.com` | B2B collaborator from another org or personal Microsoft account |

### How Users and Groups Are Organised

Entra ID is the single source of truth for identities. Users sit in the Entra ID tenant and are organised into **Security Groups** (for access control) or **Microsoft 365 Groups** (for collaboration). RBAC assignments are then made against those groups at any level of the Azure management hierarchy.

**Key design principles:**

- **Assign RBAC to Security Groups, not individual users.** Entra ID Security Groups are the unit of access management — add or remove people from the group rather than adding or removing role assignments.
- **Use Managed Identities for every Azure workload.** App Services, VMs, AKS pods, Logic Apps — all support Managed Identity. The platform handles credential rotation; you never see or store a secret.
- **Assign at the narrowest scope.** Contributor at a Resource Group is safer than Contributor at a Subscription. Use Management Groups only for policies and roles that genuinely need to apply to every subscription (e.g. org-wide Reader for auditors).
- **Separate duty groups.** Create distinct groups for `deploy` (Contributor), `read` (Reader), and `secrets` (Key Vault Secrets User) rather than giving Contributor access to everyone who needs to read logs.

<Callout type="tip">
**Entra ID Privileged Identity Management (PIM)** lets you make role assignments **just-in-time** — a user requests elevation, a manager approves, and the role activates for a fixed window (e.g. 1 hour). All elevation events are logged. Use PIM for any `Owner` or `Contributor` assignment at the Subscription level.
</Callout>

## Azure RBAC: Implementing Least Privilege

The security-foundations lesson defines **least privilege** as the principle. Azure RBAC enforces it via role assignments across the management hierarchy.

### RBAC Scope Hierarchy

```mermaid
flowchart TD
    accTitle: RBAC Scope Hierarchy
    accDescr: Flowchart (td) showing Management Group Widest scope Inherits to all below, Subscription Inherits from MG, Resource Group Inherits from Subscription
    classDef step fill:none,stroke:#a855f7,color:#a855f7,rx:8,ry:8
    classDef ok   fill:none,stroke:#22c55e,color:#22c55e,rx:8,ry:8
    linkStyle default stroke-width:2.5px

    MG["Management Group<br/>Widest scope<br/>Inherits to all below"]:::step
    SUB["Subscription<br/>Inherits from MG"]:::step
    RG["Resource Group<br/>Inherits from Subscription"]:::step
    RES["Resource<br/>Most granular"]:::ok

    ROLE["Role Assignment<br/>= Identity + Role Definition + Scope"]:::ok

    MG --> SUB --> RG --> RES
    ROLE -- "Applied at any level<br/>Inherits downward" --> MG
```

**Built-in roles — the ones you'll use most often:**

| Role | What it can do |
|---|---|
| **Owner** | Full control + manages access. Reserve for platform team. |
| **Contributor** | Create/manage resources, cannot manage access |
| **Reader** | Read-only. Assign to auditors, developers needing read access. |
| **User Access Administrator** | Manage access only (assign roles) — rare |
| **Resource-specific built-in** | e.g. `Key Vault Secrets User`, `Storage Blob Data Reader` — always prefer these over Contributor |

<Callout type="warning">
**Avoid assigning Contributor at the Subscription level to individuals.** Use Contributor at the Resource Group level, scoped to specific workloads. For elevated access, use Privileged Identity Management (PIM) for just-in-time activation with approval and time limits.
</Callout>

The default is **implicit deny** — a user with no role assignments cannot do anything. Like AWS IAM, explicit deny assignments (deny assignments) override all allows.

## Managed Identities: Eliminating Credentials from Services

The security principle: **machine-to-machine authentication should use rotating, short-lived credentials — never stored secrets**.

Azure Managed Identities give every Azure resource a service principal in Entra ID — managed entirely by Azure.

```mermaid
flowchart TD
    accTitle: Managed Identities: Eliminating Credentials from Services
    accDescr: Flowchart (td) showing VM / Function App / Container App, Managed Identity Service Principal, RBAC Role Assignment on target resource
    classDef step fill:none,stroke:#a855f7,color:#a855f7,rx:8,ry:8
    classDef ok   fill:none,stroke:#22c55e,color:#22c55e,rx:8,ry:8
    linkStyle default stroke-width:2.5px

    VM["VM / Function App / Container App"]:::step
    ENTRA_ID["Entra ID<br/>(Managed Identity<br/>Service Principal)"]:::step
    RBAC_GRANT["RBAC Role Assignment<br/>on target resource"]:::ok
    TARGET["Key Vault / Storage / SQL<br/>(target resource)"]:::ok
    TOKEN["Access token<br/>(short-lived, auto-renewed)"]:::ok

    VM -- "Requests token via<br/>IMDS endpoint" --> ENTRA_ID
    ENTRA_ID -- "Validates identity" --> TOKEN
    TOKEN --> VM
    VM -- "Presents token" --> TARGET
    ENTRA_ID --> RBAC_GRANT --> TARGET
```

**System-assigned vs user-assigned:**

| | System-assigned | User-assigned |
|---|---|---|
| Lifecycle | Tied to the resource — deleted with it | Independent, can be shared across resources |
| Use when | Single resource, unique identity | Multiple resources sharing same permissions (e.g. all workers in a job) |

```bash
# Assign a role to a managed identity via CLI
az role assignment create \
  --assignee-object-id $(az vm show -g rg-prod -n myvm --query identity.principalId -o tsv) \
  --role "Key Vault Secrets User" \
  --scope "/subscriptions/$SUB_ID/resourceGroups/rg-prod/providers/Microsoft.KeyVault/vaults/prod-kv"
```

## NSGs: Default-Deny Network Segmentation

The security-foundations principle: **default-deny segmentation with explicit allows**. NSGs implement this for Azure VNets.

### NSG Priority Evaluation

```mermaid
flowchart TD

    accDescr: Flowchart (td) showing Inbound packet arrives, lower higher priority, Rule 100: Allow HTTPS 443 from specific source
    classDef step fill:none,stroke:#a855f7,color:#a855f7,rx:8,ry:8
    classDef ok   fill:none,stroke:#22c55e,color:#22c55e,rx:8,ry:8
    classDef fail fill:none,stroke:#ef4444,color:#ef4444,rx:8,ry:8
    classDef decesion fill:none,stroke:#f59e0b,color:#f59e0b,rx:8,ry:8
    linkStyle default stroke-width:2.5px

    PACKET["Inbound packet arrives"]:::step
    RULES["NSG rules evaluated<br/>in priority order (lower # = higher priority)"]:::step
    R100{"Rule 100:<br/>Allow HTTPS 443<br/>from specific source?"}:::decesion
    R200{"Rule 200:<br/>Allow specific<br/>application port?"}:::decesion
    R65500{"Rule 65500:<br/>Deny All<br/>(default rule, always present)"}:::decesion
    ALLOW["fa:fa-check Allow and stop evaluating"]:::ok
    DENY_DEF["fa:fa-xmark Deny (default)"]:::fail

    PACKET --> RULES
    RULES --> R100
    R100 -- "Match" --> ALLOW
    R100 -- "No match" --> R200
    R200 -- "Match" --> ALLOW
    R200 -- "No match" --> R65500
    R65500 --> DENY_DEF
```

**NSG rule structure:**

| Field | Example | Notes |
|---|---|---|
| Priority | 100 | 100–4096; lower = evaluated first |
| Direction | Inbound / Outbound | Separate rule sets |
| Source | IP, CIDR, Service Tag, or Application Security Group |
| Port | 443, 8080–8090, * | |
| Protocol | TCP, UDP, Any | |
| Action | Allow / Deny | |

**Application Security Groups (ASGs)** are Azure's equivalent to AWS Security Group source references — instead of IP ranges, you reference a named group of VMs:

```
Load Balancer backend pool → tagged with ASG: asg-web
App servers → tagged with ASG: asg-app  
Databases → tagged with ASG: asg-db

NSG rule: Allow 8080 from asg-web to asg-app
NSG rule: Allow 1433 from asg-app to asg-db
```

<Callout type="info">
Apply NSGs at both the **subnet level** (broad tier enforcement) and the **NIC level** (per-VM fine-grained rules) when compliance requires micro-segmentation. Both NSGs are evaluated — the subnet NSG first, then the NIC NSG.
</Callout>

**Private Endpoints** — a key Azure security pattern: instead of accessing Key Vault or Storage over the public internet, deploy a Private Endpoint inside your VNet. The service gets a private IP in your subnet, and DNS resolves to that private IP automatically via Private DNS Zones.

## Key Vault: Secrets, Keys, and Certificates

The security principle: **never store secrets in code, config, or environment variables**. Key Vault centralises and audits secret access.

```mermaid

    KEY["Key: CMK for<br/>Storage/Disk encryption"]:::fail
    CERT["Certificate:<br/>TLS cert lifecycle"]:::ok

    APP -- "Authenticates via" --> MI
    MI -- "RBAC: Key Vault Secrets User" --> KV
    KV --> SECRET & KEY & CERT
    KV --> AUDIT
```

**Key Vault access model:**

```bash
# Grant a managed identity access to read secrets only
az keyvault set-policy \
  --name prod-kv \
  --object-id $MANAGED_IDENTITY_PRINCIPAL_ID \
  --secret-permissions get list

# Preferred: use RBAC model (not vault access policies)
az role assignment create \
  --role "Key Vault Secrets User" \
  --assignee $MANAGED_IDENTITY_PRINCIPAL_ID \
  --scope /subscriptions/$SUB/resourceGroups/rg/providers/Microsoft.KeyVault/vaults/prod-kv
```

<Callout type="tip">
**Use the Key Vault RBAC permission model, not vault access policies.** RBAC integrates with Entra ID PIM (just-in-time elevation), supports audit via activity logs, and is consistent with the rest of Azure's access control model. Vault access policies are the legacy model.
</Callout>

## Azure Monitor + Defender for Cloud: Traceability

The security principle: **every privileged action must be traceable and anomalies must trigger alerts**.

```mermaid

    EVENTS["Azure Activity Log<br/>(every ARM operation: who, what, when, status)"]:::step
    DIAG["Diagnostic Settings<br/>(resource-level logs: NSG flows, Key Vault access, etc.)"]:::step
    LAW["Log Analytics Workspace<br/>Centralised log store"]:::ok
    SENTINEL["Microsoft Sentinel<br/>(SIEM/SOAR — correlates across all logs)"]:::ok
    DEFENDER["Defender for Cloud<br/>Security posture + threat alerts"]:::ok
    ALERTS["Azure Monitor Alerts<br/>(metric or log-based alarms → Action Groups)"]:::fail

    EVENTS --> LAW
    DIAG --> LAW
    LAW --> SENTINEL & ALERTS
    DEFENDER --> LAW & SENTINEL
```

**High-priority alerts to configure:**

| Alert | Detection target |
|---|---|
| Subscription Owner assignment | Privilege escalation |
| NSG rule opens `0.0.0.0/0` any port | Accidental exposure |
| Key Vault secret accessed from new IP | Credential harvesting |
| Bulk delete of resources | Destructive action / ransomware |
| MFA-less sign-in to privileged role | Credential compromise |
| `az login` from unusual location | Account takeover indicator |

**Defender for Cloud** (free tier = Secure Score + recommendations; paid = per-resource threat detection):
- Secure Score shows percentage of controls met — use it as a living checklist
- Enable at the subscription level immediately after creation
- Enable enhanced security for any subscription running production workloads

## Azure Security Hardening Script

```bash
#!/bin/bash
# Baseline security for a new Azure subscription

SUB_ID=$(az account show --query id -o tsv)

# 1. Enable Defender for Cloud (basic plan)
az security auto-provisioning-setting update \
  --name mma --auto-provision on

# 2. Enable diagnostic settings → Log Analytics
az monitor diagnostic-settings create \
  --name "sub-activity-to-law" \
  --subscription $SUB_ID \
  --workspace $LAW_ID \
  --logs '[{"category":"Administrative","enabled":true},{"category":"Security","enabled":true},{"category":"Alert","enabled":true}]'

# 3. Assign Security Reader to security team group
az role assignment create \
  --role "Security Reader" \
  --assignee $SECURITY_GROUP_ID \
  --scope /subscriptions/$SUB_ID

# 4. Enable soft-delete on Key Vault (prevent accidental permanent deletion)
az keyvault update --name prod-kv \
  --enable-soft-delete true \
  --enable-purge-protection true
```

## Cost Implications

Security services sit at the widest range of Azure pricing — some are completely free, while others carry significant per-resource monthly costs that compound with scale.

| Service | Cost Model | Approximate Cost |
|---|---|---|
| **RBAC / Entra ID (built-in roles)** | Free | Free |
| **Managed Identities** | Free | Free |
| **NSGs and ASGs** | Free | Free |
| **Azure Key Vault (Standard)** | Per-operation | $0.03 per 10k secret operations |
| **Key Vault HSM-backed keys** | Per key + operations | $1.00/key/month + $0.015/10k ops |
| **Microsoft Defender for Cloud (free tier)** | Free | CSPM recommendations only |
| **Defender for Servers Plan P2** | Per server-hour | ~$15/server/month |
| **Log Analytics ingestion** | Per GB ingested | ~$2.76/GB (first 5 GB/day free) |
| **Log Analytics retention** | Per GB/month after 31 days | $0.10/GB/month |

<Callout type="warning" title="Defender for Cloud Adds Up Quickly">
Defender for Servers Plan P2 at ~$15/server/month costs $1,500/month on 100 VMs. Enable it selectively on internet-facing or data-tier VMs first, not uniformly across every dev/test instance. Use the free tier CSPM to identify misconfigurations — most findings are actionable without enabling paid Defender plans.
</Callout>

<Callout type="tip" title="Log Analytics Ingestion is the Silent Cost">
Activity logs and diagnostic settings routed to Log Analytics can ingest gigabytes per day across a busy subscription. Set a **daily ingestion cap** on the workspace during initial setup and review actual ingestion after 7 days before removing it for production. Data Access audit logs are especially verbose.
</Callout>

<KeyPoints title="Azure Security Checklist">
- RBAC on groups at Resource Group scope — not Contributor at Subscription to individuals
- Use PIM for elevated access — just-in-time, time-limited, approval-gated
- Managed Identities on every service — zero stored credentials
- NSGs: Rule 65500 DenyAll always present; explicitly open only what's needed; use ASGs for source/dest references
- Three-tier VNet: public (App GW/LB), private-app (VMSS/containers), private-data (DB + Key Vault via Private Endpoint)
- Key Vault RBAC model (not vault access policies); Managed Identity → Key Vault Secrets User only
- Enable Defender for Cloud on every subscription; target Secure Score improvements weekly
- Activity Log + Diagnostic Settings → Log Analytics workspace; alert on Owner assignments, NSG 0.0.0.0/0 opens, Key Vault access from new IPs
</KeyPoints>
