## IAM: Implementing Least Privilege in AWS

The security-foundations lesson defines **least privilege** as the principle. IAM is the AWS tool that enforces it.

### IAM Policy Anatomy

```mermaid
flowchart TD
    accTitle: IAM Policy Anatomy
    accDescr: Flowchart (td) showing JSON document, Effect: Allow Deny, roles/users/services
    classDef step fill:none,stroke:#a855f7,color:#a855f7,rx:8,ry:8
    classDef ok   fill:none,stroke:#22c55e,color:#22c55e,rx:8,ry:8
    linkStyle default stroke-width:2.5px

    POLICY["IAM Policy (JSON document)"]:::step
    STATEMENT["Statement block(s)"]:::step
    EFFECT["Effect:<br/>Allow | Deny"]:::ok
    PRINCIPAL["Principal:<br/>Who this applies to<br/>(roles/users/services)"]:::step
    ACTION["Action:<br/>What operation<br/>(s3:GetObject, ec2:RunInstances)"]:::step
    RESOURCE["Resource:<br/>ARN of specific resource<br/>or * for all"]:::step
    CONDITION["Condition (optional):<br/>Context constraints<br/>(IP range, MFA required, time)"]:::step

    POLICY --> STATEMENT
    STATEMENT --> EFFECT & PRINCIPAL & ACTION & RESOURCE & CONDITION
```

**Example — least-privilege S3 read for a specific Lambda:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowLambdaReadSpecificBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-data",
        "arn:aws:s3:::my-app-data/*"
      ]
    }
  ]
}
```

Note what is **not** in this policy: no `s3:PutObject`, no `s3:DeleteObject`, no `*` wildcard on actions, and the resource is scoped to a single bucket — not `arn:aws:s3:::*`.

### IAM Policy Evaluation

```mermaid
flowchart TD

    accDescr: Flowchart (td) showing Access request arrives, Explicit Deny in any policy, Explicit Allow in any policy
    classDef step fill:none,stroke:#a855f7,color:#a855f7,rx:8,ry:8

    linkStyle default stroke-width:2.5px

    REQ["Access request arrives"]:::step
    DENY_EX{"Explicit Deny<br/>in any policy?"}:::decesion
    ALLOW_EX{"Explicit Allow<br/>in any policy?"}:::decesion
    ORG_SCP{"Allowed by<br/>Org SCP?"}:::decesion
    BOUNDARY{"Within permission<br/>boundary?"}:::decesion
    ALLOW["fa:fa-check Allow"]:::ok
    DENY["fa:fa-xmark Deny"]:::fail

    REQ --> DENY_EX
    DENY_EX -- "Yes" --> DENY
    DENY_EX -- "No" --> ORG_SCP
    ORG_SCP -- "No" --> DENY
    ORG_SCP -- "Yes" --> BOUNDARY
    BOUNDARY -- "No" --> DENY
    BOUNDARY -- "Yes" --> ALLOW_EX
    ALLOW_EX -- "Yes" --> ALLOW
    ALLOW_EX -- "No (implicit deny)" --> DENY
```

**Explicit Deny wins over everything** — even if an Allow exists elsewhere in the policy chain. This is used to enforce guardrails: an SCP that denies `ec2:RunInstances` in non-approved regions cannot be overridden by any IAM policy in a child account.

### IAM Roles: No Credentials on Instances

```mermaid
flowchart TD
    accTitle: IAM Roles: No Credentials on Instances
    accDescr: Flowchart (td) showing or Lambda, ECS task, Instance Metadata Service http://169.254.169.254, Security Token Service
    classDef step fill:none,stroke:#a855f7,color:#a855f7,rx:8,ry:8
    classDef ok   fill:none,stroke:#22c55e,color:#22c55e,rx:8,ry:8
    linkStyle default stroke-width:2.5px

    EC2["EC2 Instance<br/>(or Lambda, ECS task)"]:::step
    META["Instance Metadata Service<br/>http://169.254.169.254"]:::step
    STS["AWS STS<br/>(Security Token Service)"]:::step
    ROLE["IAM Role<br/>(attached to instance)"]:::step
    CREDS["Temporary credentials<br/>AccessKeyId + SecretKey + SessionToken<br/>Auto-rotated every ~1 hour"]:::ok

    EC2 -- "Requests credentials" --> META
    META -- "Calls AssumeRole" --> STS
    STS -- "Assumes" --> ROLE
    STS -- "Returns" --> CREDS
    CREDS --> EC2
```

<Callout type="warning">
**Never put credentials in source code, AMIs, or environment variables committed to git.** Attach an IAM Role to the instance or task — the AWS SDK automatically picks up temporary credentials from the metadata service. There's nothing to rotate, nothing to leak, and the credentials expire automatically.
</Callout>

## Security Groups: Default-Deny Network Segmentation

The security-foundations lesson defines **default-deny segmentation** as the principle. Security Groups implement it at the network layer.

### Three-Tier Security Group Chaining

```mermaid
flowchart TD
    accTitle: Three-Tier Security Group Chaining
    accDescr: Flowchart (td) showing Internet, redirect, App Server Security Group Inbound: 8080 from ALB_SG only Outbound: 5432 to DB_SG
    classDef step fill:none,stroke:#a855f7,color:#a855f7,rx:8,ry:8
    classDef ok   fill:none,stroke:#22c55e,color:#22c55e,rx:8,ry:8
    classDef fail fill:none,stroke:#ef4444,color:#ef4444,rx:8,ry:8
    linkStyle default stroke-width:2.5px

    INTERNET["fa:fa-globe Internet"]:::step
    ALB_SG["ALB Security Group<br/>Inbound: 443 from 0.0.0.0/0<br/>Inbound: 80 from 0.0.0.0/0 (redirect)"]:::ok
    APP_SG["App Server Security Group<br/>Inbound: 8080 from ALB_SG only<br/>Outbound: 5432 to DB_SG"]:::ok
    DB_SG["Database Security Group<br/>Inbound: 5432 from APP_SG only<br/>No internet path"]:::fail

    INTERNET --> ALB_SG --> APP_SG --> DB_SG
```

The key property: the App SG **references the ALB SG as the source** — not an IP range. This means:
- If the load balancer's IP changes, the rule still works (it tracks the group, not the IP)
- Even if someone from the internet spoofs an IP, they can't reach the app tier — only the actual ALB can initiate the connection
- The database is completely unreachable from the internet — there is no route or security group rule that permits it

### Security Group Rules Reference

```json
// App tier SG inbound rule — source is another security group
{
  "IpPermissions": [{
    "FromPort": 8080,
    "ToPort": 8080,
    "IpProtocol": "tcp",
    "UserIdGroupPairs": [{
      "GroupId": "sg-0alb12345678",
      "Description": "Allow traffic from ALB only"
    }]
  }]
}
```

## KMS: Encryption at Rest

The security-foundations principle: **encrypt sensitive data at rest**. KMS is the AWS service that provides managed keys.

```mermaid
flowchart TD
    accTitle: KMS: Encryption at Rest
    accDescr: Flowchart (td) showing S3 PUT, EBS write, RDS write, DEK, CMK
    classDef step fill:none,stroke:#a855f7,color:#a855f7,rx:8,ry:8
    classDef ok   fill:none,stroke:#22c55e,color:#22c55e,rx:8,ry:8
    classDef fail fill:none,stroke:#ef4444,color:#ef4444,rx:8,ry:8
    linkStyle default stroke-width:2.5px

    DATA["Plaintext data<br/>(S3 PUT, EBS write, RDS write)"]:::step
    DEK["Data Encryption Key (DEK)<br/>Generated per object/volume<br/>Symmetric AES-256"]:::step
    KMS["AWS KMS<br/>Customer Master Key (CMK)"]:::ok
    ENVELOPE["Envelope Encryption:<br/>DEK encrypted with CMK<br/>Encrypted DEK stored alongside data"]:::ok
    STORED["Encrypted data + encrypted DEK<br/>(CMK never leaves KMS)"]:::fail

    DATA --> DEK
    DEK -- "Encrypts" --> DATA
    KMS -- "Encrypts DEK" --> ENVELOPE
    ENVELOPE --> STORED
```

**KMS key types:**

| Type | Who manages rotation | Use when |
|---|---|---|
| **AWS managed key** (`aws/s3`, `aws/ebs`) | AWS auto-rotates annually | Low compliance requirements, simple use |
| **Customer managed key (CMK)** | You control rotation, policy, grants | Regulatory requirements, cross-account access, key deletion control |
| **External key material** | You provide and manage | HSM or BYOK compliance requirements |

<Callout type="tip">
**Enable automatic CMK rotation.** AWS rotates the key material annually when enabled, without re-encrypting existing data (uses key versions). Disable rotation only if audit requirements mandate manual rotation.
</Callout>

## CloudTrail: Implementing Traceability

The security-foundations principle: **every privileged action must be traceable**. CloudTrail records every API call in your AWS account.

```mermaid

    classDef step fill:none,stroke:#a855f7,color:#a855f7,rx:8,ry:8
    classDef ok   fill:none,stroke:#22c55e,color:#22c55e,rx:8,ry:8
    classDef fail fill:none,stroke:#ef4444,color:#ef4444,rx:8,ry:8
    linkStyle default stroke-width:2.5px

    API["Any AWS API call<br/>(Console, CLI, SDK, service)"]:::step
    CT["CloudTrail<br/>Captures: who, what, when, from where<br/>Includes: user/role ARN, IP, request params, response"]:::step
    S3["S3 Bucket<br/>(CloudTrail logs — versioned, encrypted, object lock)"]:::ok
    CWL["CloudWatch Logs<br/>(real-time alerting)"]:::step
    ALERT["CloudWatch Alarm<br/>e.g. root account used<br/>console login without MFA<br/>Security Group opened to 0.0.0.0/0"]:::fail
    SNS["SNS → Email / PagerDuty<br/>Security alert"]:::fail

    API --> CT
    CT --> S3
    CT --> CWL --> ALERT --> SNS
```

**High-priority CloudTrail alerts to configure on day one:**

| Event | Why alert |
|---|---|
| `ConsoleLogin` without MFA | Credential compromise indicator |
| `Root` account activity | Root should almost never be used |
| `AuthorizationFailure` spike | Scanning or misconfigured automation |
| `DeleteTrail` / `StopLogging` | Attacker covering tracks |
| Security Group `AuthorizeSecurityGroupIngress` `0.0.0.0/0` | Accidental exposure |

<KeyPoints title="AWS Security Checklist">
- IAM policies: explicit Allow on minimum actions + specific resource ARNs; never wildcard on sensitive services
- Explicit Deny wins over all — use SCPs at Org level for hard guardrails
- IAM Roles on EC2/Lambda/ECS — temporary credentials auto-rotated, nothing to leak
- Security Groups: default deny, source-group chaining between tiers (Internet → ALB SG → App SG → DB SG)
- Three-tier VPC: public for LBs, private-app for servers (NAT outbound), private-data for databases (no internet)
- KMS CMK for sensitive data; enable automatic key rotation
- CloudTrail in all regions with S3 archive + CloudWatch alarms on root use, MFA-less logins, trail deletion
- GuardDuty + Security Hub for automated threat detection and compliance posture
</KeyPoints>
